# SocTalk Launchpad — design draft

**Status:** brainstorm; not approved, not implemented. Captured here so a
codex review pass can interrogate the DX/UX before any code lands.

> **Reviewer brief — read first.** This is a product/UX design draft, not
> code. Review it from a DX/UX angle, biased toward the perspective of
> an MSSP technical lead and a tenant IT contact who have never seen
> SocTalk before. Focus areas:
>
> 1. Wizard flow — where could an operator get stuck, surprised, or have
>    to back up? Are inputs requested in an order that lets the operator
>    answer them confidently?
> 2. The human-handoff moment — MSSP admin runs a one-line installer on a
>    tenant host. Is the trust/consent gesture clear? Can it fail
>    gracefully? What does the tenant IT contact actually see, and is
>    that what we want them to see?
> 3. Failure & recovery — what does the operator need when the agent
>    install hangs, the MSSP hypervisor creds are wrong, the tenant uses
>    a Wazuh version that doesn't match, the LLM key is invalid, etc.?
> 4. Defaults — is silently defaulting to bare-host doing the operator a
>    favor or hiding an important choice? Same question for other
>    invisible defaults baked into the plan.
> 5. Lifecycle moments — kicking off, mid-pilot, powering down. Does
>    each have an honest, testable answer?
> 6. What's MISSING that a real MSSP pilot would hit on day one and
>    we haven't accounted for?
>
> Be opinionated. Brainstorm-stage feedback like *"this idea is wrong"*,
> *"fold this into that"*, *"invert the order"*, or *"name this thing
> something else"* is welcome. The 10 numbered Open Questions at the
> bottom of the doc are explicit invitations to push back.

## Problem

MSSPs evaluating SocTalk run a pilot for 1–3 of their customers. Both MSSP
and tenant environments are typically on-premise, often VMware ESX, behind
asymmetric firewalls (no inbound to tenant, often no inbound to MSSP either).
Today there's no orchestration story: the MSSP follows a long manual checklist
on the MSSP side (k3s, soctalk-system chart, ingress, TLS, LLM key, …) and
then repeats most of it on each tenant side, plus a Tailscale or VPN dance
in between. Time-to-first-chat is several days and most of it is operator
toil — not product evaluation.

## Goal

A self-contained, throwaway "Launchpad" VM that an MSSP imports once per
pilot. It walks the MSSP through a single wizard, drives all the
provisioning on both sides, and hands back a working multi-tenant SocTalk
install with the AI SOC chat answering questions about real Wazuh data
within an hour or two of pilot kickoff. The MSSP touches a hypervisor
console exactly twice: once to import the Launchpad itself, once per
tenant to run a one-line installer on a tenant-network Linux box.

## Shape

### Artifact

`soctalk-launchpad` — a new VM image (~500 MB), published alongside
`soctalk-system` / `soctalk-tenant` / `soctalk-adapter` /
`soctalk-orchestrator` / `soctalk-linux-ep` to ghcr.io as
`ghcr.io/soctalk/soctalk-launchpad:<ver>` and as
`soctalk-launchpad-<ver>.{qcow2,vmdk,vhdx,vhd,raw}.xz` attached to the
GitHub Release. Same packer pipeline as the existing demo image, separate
output filename.

### Lifecycle

- **Launchpad VM** — alive for the pilot duration (30-90 days),
  archived/decommissioned at pilot end. NOT a long-term ops console.
- **Launchpad agents** (one per tenant) — installed by the MSSP admin on
  a Linux box inside the tenant's network at pilot start. Default
  behaviour: agent self-removes after the tenant install completes a
  health check. MSSP can opt to keep agents alive for the pilot window
  for post-install ops (`helm upgrade`, log fetch, redeploy).
- **`soctalk-system` and `soctalk-tenant` VMs/chart installs** — the
  actual deliverable; long-lived; the Launchpad created them but doesn't
  own them.

### Transport

Tailscale (or Headscale for self-hosted). The Launchpad VM joins an
MSSP-owned tailnet at first boot. Each tenant agent joins the same tailnet
with a tenant-tagged auth key. ACL stanza generated by the Launchpad
restricts each tenant agent to inbound-only from `tag:launchpad`; tenant
agents cannot reach each other. The whole transport runs over outbound
UDP 443 from each endpoint — no inbound firewall asks at either side.

### Two install modes per tenant

- **Bare-host (default).** The Linux box where the MSSP admin ran the
  agent installer IS the tenant SocTalk box. Agent installs k3s, helm,
  the `soctalk-tenant` chart on the same host. If the tenant doesn't
  already have Wazuh, the chart installs it too (`poc` profile). Easiest
  for tenants — IT spins up any Ubuntu VM however they normally do,
  MSSP admin runs the one-line install, agent finishes the job.
- **Appliance (opt-in).** Agent uses the tenant's hypervisor API
  (govmomi / proxmox-api / libvirt) to provision a fresh `soctalk-tenant`
  VM from a downloaded VMDK and configure it via cloud-init. Requires
  the tenant to give the agent hypervisor credentials. Use case: tenants
  who explicitly want the appliance shape and are fine with that ask.

### Asymmetric reach

- **MSSP side:** Launchpad has the MSSP's hypervisor credentials. It uses
  the hypervisor API directly to import the `soctalk-system` VMDK, boot
  it, drive cloud-init, wait for tailnet join, run the existing setup
  wizard programmatically (skipping its UI, since Launchpad has the
  answers). MSSP-side hypervisors targeted in v1: vSphere/ESX, Proxmox,
  KVM/libvirt. Hyper-V deferred (Windows automation rabbit hole).
- **Tenant side:** Launchpad has no credentials, no network reach
  pre-tailnet. MSSP admin manually opens a tenant-network shell (existing
  managed-customer access, screenshare with tenant IT, on-site, RMM tool
  — whichever fits the relationship) and runs:

  ```bash
  curl -sfL https://launchpad.<tailnet>.ts.net/agent?token=<one-time-token> | sudo sh
  ```

  Installer: downloads agent binary, installs Tailscale, joins tailnet
  with tenant-tagged auth key, registers with Launchpad, starts as
  systemd service. From then on, Launchpad drives via the agent.

### Trust model

- Embedded token in the installer URL is single-use, expires 24h, bound
  to a specific tenant slug.
- Launchpad's pilot-private-key signs every command sent to agents;
  agents reject unsigned commands. Key generated when Launchpad first
  boots; revoking it kills every agent.
- Tailscale ACL: `tag:launchpad → tag:tenant-agent-*` only.
- Kill switch: per-tenant "Remove" button in Launchpad UI drops the ACL
  entry, forces agent logout, removes the tenant tag.
- Audit log: every command + every agent response streamed to Launchpad's
  local journal; downloadable as part of pilot handoff.

### Launchpad wizard pages (sketch)

The Launchpad VM hosts a web UI on `:8443` (same convention as the
existing per-VM setup wizard, so the experience is familiar to anyone
who's seen SocTalk).

1. **Welcome + supported hypervisors.** Lists what the MSSP side can run
   on (vSphere / Proxmox / KVM). Tenant-side selection happens per tenant
   later. Sets MSSP-side hypervisor; everything that follows uses it.
2. **MSSP hypervisor creds.** vCenter URL / Proxmox URL / libvirt URI +
   login. Live "test connection" button.
3. **Tailscale.** Auth token (or Headscale URL + admin key). Launchpad
   joins tailnet now so the rest of the wizard can verify connectivity
   at each step.
4. **MSSP identity + admin.** MSSP display name, slug, admin email +
   password, LLM provider + API key. Same fields as the existing
   `soctalk-system` setup wizard, asked here so Launchpad can drive
   that wizard programmatically.
5. **Tenants.** Repeatable form per tenant: name, contact email,
   install mode (bare-host / appliance), "tenant has existing Wazuh?"
   toggle + creds if yes. List can be added to / edited until you
   click Go.
6. **Review + Go.** One-screen summary of every decision; "Start" button
   begins the run.
7. **Run.** Per-VM and per-tenant status streamed live. MSSP-side
   progress is fully automated. Tenant-side progress is gated on the MSSP
   admin running the agent installer (the wizard shows the one-line
   command per tenant, with a "copy" button; tenants show as "Waiting
   for agent" until they call home, then progress through "Installing",
   "Configuring", "Verifying").
8. **Done.** MSSP dashboard URL + admin login. Downloadable handoff
   package: `pilot-handoff.tar.gz` containing bundles, ACL, audit log,
   each tenant's connection details. "You can power this VM off now"
   (or "Keep alive for ops window" toggle).

### What the Launchpad VM ships with

- Web UI (Go binary + embedded SvelteKit assets, same stack as the
  existing setup wizard for code reuse).
- govmomi (vSphere), proxmox-api-go (Proxmox), libvirt-go (KVM).
- Tailscale daemon (the Launchpad itself joins the tailnet).
- Image cache for `soctalk-system-<ver>.vmdk/qcow2/raw` etc. Downloads
  on first boot from the matching ghcr.io release; cached on local disk
  for reuse across tenants. Embedded would be too large; downloads also
  pin pilot version to product version automatically.
- Bundle generator + cloud-init seed builder.
- Launchpad agent binary cache (downloaded once, served back over
  tailnet to each tenant agent installer URL).

## Open questions to interrogate

1. **The wizard's question order.** MSSP hypervisor creds in step 2 are
   asked BEFORE the operator sees a connectivity test happen against
   Tailscale; if Tailscale fails in step 3 they wasted time on step 2.
   But Tailscale can't realistically be tested without first knowing
   the MSSP side will boot. Question: should the wizard restructure so
   Tailscale verification happens before any MSSP-side work?

2. **Bare-host vs appliance default.** The plan says bare-host is the
   default and appliance is opt-in. Is the wizard surfacing that choice
   at the right friction level — is it OK to default it silently, or
   should the per-tenant form explicitly ask "How does this tenant want
   SocTalk installed?" so the MSSP is making an intentional choice rather
   than accepting a default?

3. **The agent install handoff moment.** Step 7 shows the wizard with
   tenants in "Waiting for agent" state and a copy-able install command
   per tenant. But the MSSP has to actually get that command to the
   tenant — paste in Slack, screenshare, whatever. Should the wizard
   offer to email it, generate a tenant-facing URL the contact can hit
   to see their command, or just stay out of the way and let the MSSP
   handle the human coordination?

4. **What happens when something fails?** Especially the agent install
   step (most error-prone since it crosses an org boundary). If the
   one-liner fails on the tenant box, the agent never registers, the
   wizard sits in "Waiting for agent" forever. Is there a "retry this
   tenant" affordance that regenerates the token + URL? A "skip this
   tenant" affordance that lets the rest of the pilot proceed? A
   diagnostic page the MSSP admin can show the tenant IT contact?

5. **The "you can power this VM off now" moment.** After Done, the
   Launchpad's job is conceptually finished but the agents are still
   running on tenant hosts. If the MSSP powers off the Launchpad, the
   agents have no controller. Should the wizard explicitly require the
   MSSP to either (a) tear down all agents first or (b) accept that they
   become orphaned and self-remove on next idle cycle?

6. **Re-running the wizard.** If the operator closes the browser
   mid-wizard, can they pick up where they left off, or do they restart?
   State is in the Launchpad VM's local DB, so picking up is technically
   possible. Is it worth the UI complexity, or do we just keep the
   wizard short enough that restart is fine?

7. **Discoverability of post-install ops.** Bare-host agents that stay
   alive for the pilot window can do `helm upgrade`, log fetch, restart.
   Where in the UI does the MSSP discover that capability? Embedded in
   the per-tenant detail page on the Launchpad? In the MSSP product
   dashboard once it's up (separate from the Launchpad)? Both?

8. **Naming.** `soctalk-launchpad` vs `soctalk-pilot` vs
   `soctalk-deploy` vs `soctalk-bootstrap`. Currently leaning Launchpad.
   Does the name communicate "pilot-shaped throwaway tool" or does it
   read as a longer-lived production thing the buyer might expect to
   keep using?

9. **The handoff package.** What's actually inside `pilot-handoff.tar.gz`
   that a security-minded MSSP would want? The audit log, the ACL stanza,
   per-tenant connection details, the agent removal instructions, copies
   of any signed bundles, the Launchpad's pilot-private-key (so they can
   prove provenance), the system + tenant images they used? Too much vs
   too little.

10. **Time-to-first-value claim.** Plan implicitly promises "working
    multi-tenant SocTalk + AI chat in 1-2 hours". That number assumes:
    MSSP imports Launchpad VMDK (~10 min), wizard pages 1-6 filled
    (~10 min), MSSP-side install completes (~15 min for chart pull +
    soctalk-system boot), each tenant agent install + tenant-side
    install (~15 min wall clock if MSSP admin and tenant IT are sync;
    much longer if they're async). Is this honest? Should the wizard
    surface the elapsed time + remaining estimate during step 7?
